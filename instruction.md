Checkov Scan & Kyverno Policy Enforcement  

Two policy layers are required: Checkov for IaC static analysis, and Kyverno for runtime Kubernetes admission control. Both must be enforced in the pipeline.

Write three Kyverno ClusterPolicies to your k3s cluster. Code should be in to infrastructure/kyverno-policies/. They must all be in Enforce mode (not Audit):
•	Policy 1 — no-root-containers: all containers must have securityContext.runAsNonRoot: true. Pods violating this must be rejected at admission.
•	Policy 2 — require-resource-limits: all containers must declare resources.limits.cpu and resources.limits.memory. No resource limits = rejected at admission.
•	Policy 3 — ecr-only-images: all images must come from your project's ECR registry (your-account-id.dkr.ecr.region.amazonaws.com). Images from Docker Hub or public registries must be rejected.

Replace your standard Kubernetes Deployment with an Argo Rollouts Rollout resource. The canary must be fully automated with a Prometheus-based analysis step that determines whether to promote or abort.
Required canary configuration:
•	Steps: 10% canary → pause 2 minutes → AnalysisRun → 25% → pause 2 minutes → 50% → full rollout
•	AnalysisTemplate: query Prometheus for error rate on the canary pods over a 2-minute window. If error rate > 2%, auto-abort and roll back to stable. If error rate <= 2%, auto-promote to next step.
•	The PromQL query in your AnalysisTemplate must match the metric your application actually emits — not a generic template query.
•	Set a progressDeadlineSeconds of 600. If the rollout does not progress within 10 minutes, it must abort automatically.

code of your Rollout manifest and AnalysisTemplate to k8s/rollout.yaml and k8s/analysis-template.yaml. The SLO YAML in Part 3 must reference the same PromQL query used here — the canary gate and the SLO must share a measurement definition.

Pipeline Engineering Standards
•	Fail fast and ordered: Stage 1 (secret scanning) must run before Stage 5 (build). If Stage 1 fails, Stages 2–13 must not start. Use needs/depends_on correctly — show this in the YAML.
•	OIDC only: the Argo CD sync stage must use aws-actions/configure-aws-credentials with role-to-assume. The pipeline YAML must not contain AWS_ACCESS_KEY_ID anywhere — including in comments.
•	Reusable workflows: extract the security scanning stages (1–4) into a reusable workflow (.github/workflows/security-scan.yml) that the main pipeline calls. This models how a platform team shares standards across repositories.
•	CODEOWNERS: add a CODEOWNERS file requiring senior engineer review on any change to .github/workflows/, k8s/, or infrastructure/ directories.
•	Image tag immutability: the pipeline must push with a git-SHA tag and configure the ECR repository with imageTagMutability = IMMUTABLE. The Rollout manifest must reference the SHA tag, not :latest.

Disaster Recovery Design:

This part does not require you to build a DR environment — it requires you to think like an engineer who designs one. The deliverable is a written DR design document for the Kubernetes environment you built in Parts 1–3

'If the AWS region we're deployed in has a full outage tonight, what is our recovery plan? How long until we're back and how much data do we lose?'
You have 24 hours to produce a credible written answer. Your k3s cluster, Argo CD, Kyverno policies, application images, and Terraform state all exist in a single AWS region. Nothing is cross-region replicated. Go.

DR Design Document  [15 Marks]
Write docs/dr-design.md. Use sub-headings — not prose paragraphs — for each of the following five sections:

Section 1 — Current state inventory
List every component in your deployed environment: EC2 instance (k3s), ECR images, Terraform state (S3), Kyverno policies, Argo CD configuration, application Kubernetes manifests (stored in Git). For each component, state: where it lives, what is lost in a full ap-south-2 regional failure, and whether it is currently recoverable without any additional DR setup.

Section 2 — RTO and RPO targets
Propose specific RTO and RPO targets for this service and justify them. The justification must reference the business context — is this an internal operations tool or a customer-facing SaaS? They have different tolerances. Show the calculation: at your chosen SLO target from Part 3, what is the maximum allowed downtime per month? How does that constrain your RTO?

Section 3 — Kubernetes-specific recovery strategy
This is where DR on Kubernetes differs from Lambda or traditional compute. Address each component:
•	k3s cluster: how do you rebuild it? Terraform re-applies the EC2 instance — but k3s itself, its configuration, and its state need reinstalling. What is the recovery procedure and how long does it take?
•	Kubernetes manifests: they live in Git (GitOps). What is the exact sequence to restore Argo CD and sync all applications? Include the step where Argo CD itself is not yet running.
•	Kyverno policies: they live in Git. What order do you restore them relative to application deployments — and why does order matter?
•	Container images in ECR: ECR is regional. If ap-south-2 is unavailable, your images are unavailable. What is your strategy? Cross-region replication? A backup registry? Rebuilding from source? Evaluate the trade-offs.
•	Terraform state in S3: S3 has 99.99% availability but is regional. What happens to your state file in a regional failure? Does S3 cross-region replication solve this or introduce new problems (state divergence)?

Section 4 — DR runbook (numbered, command-level)
Write a numbered, command-level recovery procedure. Not prose — numbered steps, with the actual CLI commands or Helm commands at each step. Someone who has never seen this environment must be able to execute this runbook from scratch. The runbook must end with a validation step: the exact curl command and expected response that confirms recovery was successful.
Runbook standard — non-negotiable
'Reinstall Argo CD' is not a runbook step. 'helm repo add argo https://argoproj.github.io/argo-helm && helm install argocd argo/argo-cd -n argocd --create-namespace -f k8s/argocd-values.yaml' is a runbook step. Every step must have the actual command.

Section 5 — Gap analysis and production DR investment
Your current free-tier architecture is not production-grade for DR. Name the specific gaps: ECR cross-region replication not configured, S3 state bucket cross-region replication not enabled, no automated DR drill mechanism, RTO is unmeasured (estimated, not tested). For each gap, state what you would add with a real infrastructure budget and estimate the effort.

How Section 4 is scored
The RTO/RPO calculation is the technical litmus test. 'RTO is 4 hours' without showing how you arrived at that number does not score. The runbook must have actual commands — 'reinstall the cluster' is not a runbook step. The gap analysis is worth as many marks as the runbook — identifying what is missing is as important as describing what is there.
