QUESTION 1: WHERE IS "WATCH GITHUB" SETUP?
================================================================================

THE QUESTION:
  "Where is the setup for Argo CD to watch GitHub?
   I see: Tell Argo CD: 'Watch GitHub, apply k8s/ files' what do u mean? 
   How? Where?"

THE CRYSTAL CLEAR ANSWER:

LOCATION:
  File: k8s/argocd-app.yaml
  Status: ALREADY EXISTS in your repo (you can view it right now)

WHAT'S IN THIS FILE:
  This YAML file contains ONE Kubernetes Application resource that tells
  Argo CD: "Watch this GitHub repo and sync this folder"

  Content (simplified):
    apiVersion: argoproj.io/v1alpha1
    kind: Application
    metadata:
      name: genesis-platform-api
      namespace: argocd
    spec:
      source:
        repoURL: https://github.com/papunpatra27-dot/genesis-devsecops-papun.git
        targetRevision: main
        path: k8s
      destination:
        server: https://kubernetes.default.svc
        namespace: staging
      syncPolicy:
        automated:
          prune: true
          selfHeal: true

  MEANING OF KEY LINES:
    Line "source.repoURL": 
      WHERE to watch? = Your GitHub repo
    
    Line "source.targetRevision: main": 
      WHICH branch? = main branch
    
    Line "source.path: k8s": 
      WHICH folder? = k8s/ folder
    
    Line "destination.namespace: staging": 
      WHERE to deploy? = staging namespace in k8s
    
    Line "automated": 
      WHAT to do when changes detected? = Auto-sync

HOW DOES IT WORK?

  Step 1: File k8s/argocd-app.yaml is stored in YOUR GITHUB REPO
  Step 2: You run: kubectl apply -f k8s/argocd-app.yaml
  Step 3: Kubernetes sees: "Application resource" → passes to Argo CD controller
  Step 4: Argo CD controller reads: "Watch GitHub repo, k8s folder, main branch"
  Step 5: Argo CD POLLS GitHub every 3 minutes:
    • Fetch: Latest commit from main branch
    • Read: All YAML files in k8s/ folder
    • Compare: Git state vs cluster state
    • If different: Apply changes (kubectl apply -f)

THE ANSWER TO "HOW?":
  The YAML file (k8s/argocd-app.yaml) is THE SETUP.
  This file exists in your repo.
  When you kubectl apply it, Argo CD automatically starts watching.

THE ANSWER TO "WHERE?":
  File location: k8s/argocd-app.yaml

QUESTION 3: DOES POD CIDR (10.42.0.0/16) NEED TO MATCH VPC CIDR?
================================================================================

THE QUESTION:
  "do we really need change as per vpc and ec2 ip??? or ec2 will not have ip??"

THE CRYSTAL CLEAR ANSWER:

NO. YOU DO NOT NEED TO CHANGE IT.

WHY?

Pod CIDR and VPC CIDR are TWO SEPARATE NETWORKS:

  VPC Network (AWS level):
    • Managed by AWS
    • CIDR: 10.0.0.0/16
    • EC2 instance gets IP: 10.0.1.5 (example)
    • This is BEFORE k3s installation
    • Defined in: infrastructure/modules/networking/main.tf

  Pod Network (Kubernetes level):
    • Managed by k3s
    • CIDR: 10.42.0.0/16
    • Pods get IPs: 10.42.1.5, 10.42.1.6, etc.
    • This is AFTER k3s installation
    • This is inside the EC2 OS (not in AWS)

DIAGRAM:

  Internet
    ↓
  AWS (VPC: 10.0.0.0/16)
    ↓
  EC2 instance (IP: 10.0.1.5)
    ↓ (inside EC2 OS)
  k3s container runtime
    ↓
  Pod network (10.42.0.0/16)
    ↓
  Pods (IP: 10.42.1.5, 10.42.1.6, etc)

DO THEY CONFLICT?

NO. They are completely separate.
Even if both are 10.x.x.x, they're in different network namespaces.

EXAMPLE:

  EC2 machine spec:
    Public IP: 13.201.123.45
    Private IP (from VPC): 10.0.1.5
  
  When k3s starts:
    Pod network: 10.42.0.0/16 (inside k3s)
    Pod 1: 10.42.1.5 (inside k3s)
    Pod 2: 10.42.1.6 (inside k3s)
  
  NO CONFLICT because:
    • EC2:10.0.1.5 is in AWS VPC network
    • Pod:10.42.1.5 is in k3s container network
    • Totally separate

HOW TRAFFIC REACHES PODS FROM EC2 OS:

  User sends traffic to EC2:80
    ↓
  EC2 kernel (kube-proxy iptables rules)
    ↓ (k8s networking, completely transparent)
  Pod IP:8080

  The EC2 kernel knows how to reach pod network because:
    • k3s installer (curl https://get.k3s.io | sh) sets up CNI plugin
    • CNI plugin creates Linux networking rules
    • Automatic, you don't configure it

DO YOU NEED TO KNOW THE POD CIDR?

NO. You don't need to change it.
It's automatic.

IF YOU REALLY WANTED TO CHANGE IT:

  In userdata.sh.tpl, you would run:
    curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="--cluster-cidr=10.100.0.0/16" sh -
  
  But DO NOT DO THIS. The default (10.42.0.0/16) is fine.

QUESTION 4: WHERE IS NGINX SERVICE YAML SETUP?
================================================================================

THE QUESTION:
  "where is the setup needed? is yaml already present? 
   or after helm ingress install automatically yaml will be created?"

THE CRYSTAL CLEAR ANSWER:

AFTER HELM INSTALL, YAML IS AUTOMATICALLY CREATED.
YOU DO NOT WRITE IT.

HERE'S HOW IT WORKS:

STEP 1: You run helm install command
  COMMAND:
    helm install ingress-nginx ingress-nginx/ingress-nginx \
      --namespace ingress-nginx --create-namespace

STEP 2: Helm chart is downloaded from Internet
  SOURCE: https://github.com/kubernetes/ingress-nginx
  WHERE: Helm cache (temporary)
  CONTAINS: Pre-packaged YAML templates (not in your repo)

STEP 3: Helm applies ALL pre-packaged YAML to cluster
  Helm creates (automatically):
    ✓ Namespace: ingress-nginx
    ✓ ServiceAccount: ingress-nginx
    ✓ ClusterRole: ingress-nginx
    ✓ ClusterRoleBinding: ingress-nginx
    ✓ Service: ingress-nginx-controller (type: NodePort)
    ✓ Deployment: ingress-nginx-controller
    ✓ ConfigMap: ingress-nginx-controller
    ✓ And many more resources

STEP 4: The Service YAML created by Helm
  This Service is AUTOMATICALLY generated by Helm:
    apiVersion: v1
    kind: Service
    metadata:
      name: ingress-nginx-controller
      namespace: ingress-nginx
    spec:
      type: NodePort
      ports:
        - port: 80
          targetPort: 8080
          nodePort: 30803        ← Random port assigned by k3s
        - port: 443
          targetPort: 8443
          nodePort: 32719        ← Random port assigned by k3s
      selector:
        app.kubernetes.io/name: ingress-nginx
        app.kubernetes.io/instance: ingress-nginx
        app.kubernetes.io/component: controller

WHERE IS THIS YAML STORED?

  Option 1: In Helm chart (online)
    Location: https://github.com/kubernetes/ingress-nginx/tree/main/charts/ingress-nginx
    Files: templates/controller-service.yaml
    Status: NOT in your repo (managed by upstream Helm project)
  
  Option 2: In k3s cluster (after helm install)
    Location: Inside k3s etcd database
    Status: LIVE Kubernetes resources (not in any YAML file)

DO YOU NEED TO WRITE THIS YAML?

NO. NEVER.

The ONLY thing you write is:
  COMMAND: helm install ingress-nginx ...
  
Helm does EVERYTHING ELSE automatically.

WHAT IF YOU WANT TO CUSTOMIZE IT?

Example: You want NGINX on port 8080 instead of port 80

OPTION 1: Use Helm values
  COMMAND:
    helm install ingress-nginx ingress-nginx/ingress-nginx \
      --namespace ingress-nginx --create-namespace \
      --set controller.service.ports.http=8080

  This tells Helm to customize the Service before creating it.

OPTION 2: After helm install, edit the Service
  COMMAND:
    kubectl patch service -n ingress-nginx ingress-nginx-controller \
      -p '{"spec":{"ports":[{"port":8080,"targetPort":8080}]}}'

IMPORTANT:
  These customizations are OPTIONAL.
  For your project, use the DEFAULT helm install command.
  DO NOT modify the YAML or try to apply it manually.

WHY NOT?

Because if you manually edit it, then later run:
  helm upgrade ingress-nginx ...
  
Helm will overwrite your changes with the original template.
Always use: helm values or helm upgrade, never manual kubectl edit.
